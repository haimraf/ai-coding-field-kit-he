# שער מיזוג: לפני שלוחצים Merge

## 5 שאלות לפני מיזוג

1. האם ה-diff תואם בדיוק למשימה המקורית?
2. האם קראנו את הקבצים שהשתנו, ולא רק את סיכום ה-AI?
3. האם רצה בדיקה רלוונטית או תועדה בדיקה ידנית ברורה?
4. האם אין שינוי לא מאושר ב-DB, API, הרשאות, אבטחה או פרטיות?
5. האם ה-copy וה-UX עדיין מרגישים אנושיים ומתאימים למוצר?

## רמזור החלטה

- **ירוק: אפשר למזג** — שינוי קטן, בדוק, ברור, וללא חריגה מה-scope.
- **צהוב: צריך תיקון קטן** — הכיוון נכון, אבל חסרה בדיקה, יש ניסוח חלש, או צריך להסיר שינוי צדדי.
- **אדום: לא למזג** — יש שינוי לא מאושר, סיכון נתונים/אבטחה, או diff שלא תואם לסיכום.

## דגלים אדומים

- ה-AI שינה קבצים מחוץ למשימה.
- נוספה dependency בלי אישור.
- ה-copy נהיה גנרי או פחות אנושי.
- סכמה, API או אבטחה השתנו בלי בקשה מפורשת.
- הסיכום אומר “done”, אבל ה-diff מספר סיפור אחר.

## תבנית תגובת merge

```md
עברתי על ה-PR מול המשימה המקורית.

מצב: ירוק / צהוב / אדום

מה תקין:
- [נקודה קצרה]

מה דורש טיפול לפני merge:
- [אם אין, לכתוב: אין]

בדיקות שבוצעו:
- [פקודה או בדיקה ידנית]

הערת סיכון:
- [אם אין, לכתוב: לא זוהה סיכון חריג]
```
