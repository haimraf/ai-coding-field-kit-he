# חבילת פרומפטים מעשית ל-AI Coding

## 1. תיקון UI קטן בלי חריגה מהמשימה

**מתי להשתמש:** כשיש בעיית ריווח, יישור, מצב מובייל או טקסט קצר, ואתם לא רוצים שהכלי יפתח “שיפוץ עיצובי”.

```md
יש בעיית UI קטנה ב-[מסך/קומפוננטה]: [תיאור קצר].

המטרה: לתקן רק את [ריווח/יישור/שבירת שורה/טקסט כפתור] באזור הזה.

גבולות:
- אל תשנה/י מבנה נתונים, API או לוגיקה עסקית.
- אל תוסיף/י dependency.
- אל תשכתב/י copy שלא קשור ישירות לתיקון.
- אם נראה שצריך שינוי רחב יותר, עצור/י והסבר/י למה.

בסיום דווח/י:
- אילו קבצים השתנו.
- מה בדיוק תוקן.
- איך בדקת desktop ומובייל.
- האם נשאר סיכון או משהו שלא בדקת.
```

**למה זה עובד:** הוא מגדיר תיקון צר, חוסם “שיפור” לא קשור, ומכריח דיווח שאפשר לבדוק מול ה-diff.

## 2. תיקון באג עם דיווח סיכונים

**מתי להשתמש:** כשיש באג ברור, אבל אתם רוצים להבין גם מה עלול להישבר בעקבות התיקון.

```md
יש באג: [מה המשתמש עושה] גורם ל-[תוצאה שגויה].

צפוי: [התנהגות נכונה].
בפועל: [התנהגות קיימת].

בדוק/י את הקוד הרלוונטי, הצע/י תוכנית קצרה, ואז תקן/י רק את הסיבה הסבירה ביותר לבאג.

גבולות:
- אל תעשה/י ריפקטור רחב.
- אל תשנה/י UX או copy אלא אם זה הכרחי לתיקון.
- אל תוסיף/י dependency.

בסיום כתוב/י:
- מה הייתה סיבת הבאג.
- מה השתנה.
- אילו בדיקות הרצת.
- אילו תרחישים עדיין מסוכנים או לא מכוסים.
```

**למה זה עובד:** הוא מבקש תיקון, אבל גם אבחון, בדיקות וסיכונים — בלי לפתוח משימה גדולה מדי.

## 3. סקירת PR לפני מיזוג

**מתי להשתמש:** אחרי שה-AI יצר שינוי ואתם רוצים סקירה קרה לפני merge.

```md
סקור/י את ה-PR הזה לפני מיזוג.

התמקד/י ב:
- האם ה-diff תואם למשימה המקורית.
- האם יש שינויי scope לא מוסברים.
- האם נוספו dependencies, קבצי קונפיגורציה או שינויי סכמה.
- האם יש סיכון אבטחה, פרטיות או מידע.
- האם ה-copy נשאר טבעי ולא גנרי.

אל תתקן/י עדיין. החזר/י רק סקירה קצרה עם:
- מה תקין.
- מה חייב בדיקה.
- מה צריך להסיר או לפצל.
- המלצה: ירוק / צהוב / אדום.
```

**למה זה עובד:** הוא מפריד בין סקירה לתיקון, וכך מקטין סיכוי שהכלי יוסיף עוד שינויים תוך כדי ביקורת.

## 4. מעבר QA אחרי שינוי שנוצר בעזרת AI

**מתי להשתמש:** אחרי שינוי קוד, לפני שמאמינים לסיכום “done”.

```md
בצע/י מעבר QA על השינויים האחרונים.

בדוק/י:
- אילו קבצים השתנו בפועל.
- האם השינוי נשאר בתוך המשימה.
- אילו בדיקות קיימות רלוונטיות להרצה.
- איזה תרחיש משתמש צריך לבדוק ידנית.
- האם יש סיכון מובייל, נגישות, פרטיות או נתונים.

אם אפשר להריץ בדיקות, הרץ/י אותן ודווח/י את הפקודות והתוצאה.
אם אי אפשר, הסבר/י למה ומה הייתי צריך לבדוק ידנית.

אל תעשה/י שינויים בקוד בלי לבקש אישור.
```

**למה זה עובד:** הוא מכוון את הכלי לאימות ולא לכתיבה נוספת, ומבקש פקודות אמיתיות במקום הצהרות כלליות.

## 5. ניקוי טקסט בלי להפוך אותו לגנרי

**מתי להשתמש:** כשיש copy שצריך לחדד, אבל לא רוצים שפה שיווקית או חלקה מדי.

```md
שפר/י בעדינות את הטקסט באזור [מסך/קובץ/קומפוננטה].

המטרה: שיהיה ברור, טבעי וקצר יותר.

גבולות:
- אל תשנה/י משמעות.
- אל תהפוך/י את הטון לשיווקי.
- אל תוסיף/י הבטחות, סופרלטיבים או שפה כללית.
- אל תשנה/י טקסטים שלא קשורים לאזור הזה.

החזר/י diff קטן, ובסיום פרט/י:
- אילו משפטים שונו.
- למה כל שינוי עוזר.
- איזה טקסט בכוונה נשאר כמו שהוא.
```

**למה זה עובד:** הוא נותן רשות לערוך, אבל מגן על קול אנושי ועל משמעות קיימת.

## 6. בקשת ריפקטור עם גבולות קשיחים

**מתי להשתמש:** כשצריך לסדר קוד קיים, אבל אסור לשנות התנהגות.

```md
בצע/י ריפקטור קטן ב-[קובץ/מודול] כדי לשפר קריאות ותחזוקה.

אסור לשנות התנהגות חיצונית.

גבולות:
- אל תשנה/י API ציבורי.
- אל תשנה/י copy או UI.
- אל תשנה/י סכמה, מיגרציות או מבנה נתונים.
- אל תוסיף/י dependency.
- אם צריך שינוי גדול יותר, עצור/י והצע/י אותו כמשימת המשך.

בסיום דווח/י:
- מה נהיה פשוט יותר.
- איך וידאת שהתנהגות לא השתנתה.
- אילו בדיקות רצת או חסרות.
```

**למה זה עובד:** הוא מתיר ניקוי פנימי בלבד, ומחייב הוכחה שלא נוצר שינוי מוצרי בטעות.

## 7. בדיקת אבטחה ופרטיות קצרה

**מתי להשתמש:** לפני מיזוג שינוי שנוגע לקלט משתמש, הרשאות, לוגים, API או נתונים.

```md
בצע/י sanity check לאבטחה ופרטיות על השינויים האחרונים.

חפש/י במיוחד:
- secrets או tokens שנכנסו לקוד.
- לוגים שחושפים מידע אישי או רגיש.
- קלט משתמש שלא מקבל ולידציה בסיסית.
- שינוי הרשאות או גישה לנתונים.
- שינוי API או סכמה שלא התבקש.

אל תתקן/י עדיין. החזר/י רשימה קצרה:
- אין ממצא / יש ממצא.
- חומרה: נמוכה / בינונית / גבוהה.
- איפה זה נמצא.
- מה ההמלצה המינימלית לתיקון.
```

**למה זה עובד:** הוא מחפש סיכונים מוגדרים ומונע תיקון אוטומטי שעלול להרחיב את ה-PR.

## 8. עצור ושאל כשיש אי-ודאות

**מתי להשתמש:** כשיש סיכוי שהכלי ינחש החלטת מוצר, אבטחה, סכמה או scope.

```md
אם במהלך העבודה יש אי-ודאות משמעותית, אל תנחש/י.

עצור/י ושאל/י לפני שינוי אם:
- צריך לבחור התנהגות מוצרית חדשה.
- צריך לשנות DB, סכמה, API או הרשאות.
- צריך להוסיף dependency.
- יש סיכון לחשיפת מידע אישי או secrets.
- המשימה המקורית לא מספיקה כדי לקבל החלטה בטוחה.

כשאת/ה עוצר/ת, החזר/י:
- מה לא ברור.
- למה זו החלטה מסוכנת או מוצרית.
- 2 אפשרויות פעולה קצרות.
- ההמלצה שלך, בלי לבצע שינוי עדיין.
```

**למה זה עובד:** הוא הופך עצירה להתנהגות רצויה, ולא לכישלון של הכלי.
