# חבילת פרומפטים מעשית לעבודה עם AI בקוד

המטרה של הקובץ הזה היא לתת ניסוחים קצרים שאפשר להעתיק, להתאים ולהריץ. אל תשתמשו בהם כמו קסם: הוסיפו הקשר, גבולות ובדיקות שמתאימים לריפו שלכם.

## 1. שינוי קטן וממוקד

```md
בצע/י שינוי קטן בלבד:
[מה צריך להשתנות]

גבולות:
- אל תיגע/י בקבצים מחוץ ל-[נתיב/מודול].
- אל תוסיף/י dependency חדש.
- שמור/י על התנהגות קיימת שלא קשורה למשימה.

בסיום:
- הצג/י את הקבצים ששונו.
- הסבר/י איך בדקת.
- ציין/י סיכונים או הנחות.
```

## 2. תיקון באג עם שחזור

```md
יש באג ב-[זרימה/מסך/פקודה].

צעדי שחזור:
1. [צעד]
2. [צעד]
3. [תוצאה שגויה]

תוצאה רצויה:
[מה אמור לקרות]

בקשתי:
- אתר/י את הסיבה הסבירה ביותר.
- בצע/י תיקון מינימלי.
- הוסף/י או עדכן/י בדיקה אם יש תשתית בדיקות קיימת.
- אל תשנה/י UX או API בלי לציין זאת קודם.
```

## 3. סקירת שינוי לפני PR

```md
סקור/י את ה-diff הנוכחי לפני PR.

בדוק/י במיוחד:
- שינויי scope לא מתוכננים.
- טיפול בשגיאות ומקרי קצה.
- סודות, מידע אישי או לוגים רגישים.
- קישורים, טקסטים בעברית ונגישות בסיסית.

פלט רצוי:
- בעיות שחייבות תיקון לפני מיזוג.
- הערות שכדאי לתקן אבל לא חוסמות.
- בדיקות שכדאי להריץ ידנית.
```

## 4. הוספת תיעוד קצר

```md
הוסף/י תיעוד קצר ל-[נושא].

הקהל:
[מפתחים/מוצר/צוות תמיכה/משתמשים]

הטון:
- עברית טבעית וברורה.
- בלי הבטחות שיווקיות.
- בלי להציג AI כתחליף לבדיקה או אחריות מקצועית.

גבולות:
- עד [מספר] פסקאות.
- בלי קבצי build, badges או תלות חדשה.
- קשר/י למסמכים קיימים רק אם הקישור באמת עוזר.
```

## 5. ניקוי שינוי גדול מדי

```md
ה-diff הנוכחי גדול מדי ביחס למשימה.

בקשתי:
- זהה/י אילו שינויים נחוצים למטרה המקורית.
- הצע/י מה להסיר או לפצל ל-PR נפרד.
- אל תמחק/י קוד בלי להסביר מה יוסר ולמה.

המטרה המקורית:
[תיאור קצר]
```
