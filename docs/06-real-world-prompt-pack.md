# חבילת פרומפטים מעשית לעבודה עם AI בקוד

המטרה של הקובץ הזה היא לתת ניסוחים קצרים שאפשר להעתיק, להתאים ולהריץ. אל תשתמשו בהם כמו קסם: הוסיפו הקשר, גבולות ובדיקות שמתאימים לריפו שלכם.

## 1. שינוי קטן וממוקד

```md
בצע/י שינוי קטן בלבד:
[מה צריך להשתנות]

גבולות:
- אל תיגע/י בקבצים מחוץ ל-[נתיב/מודול].
- אל תוסיף/י dependency חדש.
- שמור/י על התנהגות קיימת שלא קשורה למשימה.

בסיום:
- הצג/י את הקבצים ששונו.
- הסבר/י איך בדקת.
- ציין/י סיכונים או הנחות.
```

למה זה עובד: הוא מצמצם את מרחב הפעולה ומכריח דיווח על בדיקות וסיכונים.

## 2. תיקון באג עם שחזור

```md
יש באג ב-[זרימה/מסך/פקודה].

צעדי שחזור:
1. [צעד]
2. [צעד]
3. [תוצאה שגויה]

תוצאה רצויה:
[מה אמור לקרות]

בקשתי:
- אתר/י את הסיבה הסבירה ביותר.
- בצע/י תיקון מינימלי.
- הוסף/י או עדכן/י בדיקה אם יש תשתית בדיקות קיימת.
- אל תשנה/י UX או API בלי לציין זאת קודם.
```

למה זה עובד: צעדי שחזור ותוצאה רצויה מפחיתים ניחושים ומכוונים לתיקון קטן.

## 3. סקירת שינוי לפני PR

```md
סקור/י את ה-diff הנוכחי לפני PR.

בדוק/י במיוחד:
- שינויי scope לא מתוכננים.
- טיפול בשגיאות ומקרי קצה.
- סודות, מידע אישי או לוגים רגישים.
- קישורים, טקסטים בעברית ונגישות בסיסית.

פלט רצוי:
- בעיות שחייבות תיקון לפני מיזוג.
- הערות שכדאי לתקן אבל לא חוסמות.
- בדיקות שכדאי להריץ ידנית.
```

למה זה עובד: הוא מבקש ביקורת לפי רמות חומרה במקום תקציר כללי של השינוי.

## 4. הוספת תיעוד קצר

```md
הוסף/י תיעוד קצר ל-[נושא].

הקהל:
[מפתחים/מוצר/צוות תמיכה/משתמשים]

הטון:
- עברית טבעית וברורה.
- בלי הבטחות שיווקיות.
- בלי להציג AI כתחליף לבדיקה או אחריות מקצועית.

גבולות:
- עד [מספר] פסקאות.
- בלי קבצי build, badges או תלות חדשה.
- קשר/י למסמכים קיימים רק אם הקישור באמת עוזר.
```

למה זה עובד: הוא מגדיר קהל, טון ואורך לפני שהכלי מתחיל להרחיב.

## 5. ניקוי שינוי גדול מדי

```md
ה-diff הנוכחי גדול מדי ביחס למשימה.

בקשתי:
- זהה/י אילו שינויים נחוצים למטרה המקורית.
- הצע/י מה להסיר או לפצל ל-PR נפרד.
- אל תמחק/י קוד בלי להסביר מה יוסר ולמה.

המטרה המקורית:
[תיאור קצר]
```

למה זה עובד: הוא מחזיר את הדיון למטרה המקורית לפני שמוחקים או מפצלים עבודה.

## 6. ניקוי ניסוח בלי להפוך עברית לגנרית

```md
נקה/י את הניסוח ב-[קובץ/קטע] בלי לשנות את המשמעות.

חשוב:
- שמור/י על עברית טבעית, ישירה ולא שיווקית.
- אל תהפוך/י ניסוחים טובים לכלליים רק כדי "לשפר".
- אל תוסיף/י טענות, הבטחות או דוגמאות שלא קיימות במקור.

בסיום, הצג/י בקצרה מה שינית ולמה.
```

למה זה עובד: הוא מגן על קול מקומי ומונע ליטוש יתר שמוחק משמעות.

## 7. בדיקת אבטחה ופרטיות לפני מיזוג

```md
בצע/י בדיקת sanity לאבטחה ופרטיות לפני merge.

בדוק/י רק את ה-diff הנוכחי:
- secrets, tokens, מפתחות API או פרטי התחברות.
- מידע אישי, לוגים רגישים או דוגמאות עם נתונים אמיתיים.
- שינויי הרשאות, גישה לקבצים, רשת או מסד נתונים.

פלט רצוי:
- חוסם מיזוג: [כן/לא]
- ממצאים: [רשימה קצרה]
- מה צריך לבדוק ידנית: [אם יש]
```

למה זה עובד: הוא ממקד את הבדיקה בסיכונים שחייבים לעצור לפני merge.

## 8. לעצור ולשאול כשיש סיכון או שינוי scope

```md
אם המשימה לא ברורה, מסוכנת או דורשת שינוי scope — עצור/י ואל תבצע/י קוד.

עצור/י במיוחד אם:
- צריך לשנות API, סכמה, הרשאות או התנהגות מוצרית.
- חסר מידע כדי לבחור פתרון בטוח.
- נראה שנדרש לגעת בקבצים שלא הוגדרו במשימה.

במקום לבצע, כתוב/י:
- מה לא ברור.
- אילו אפשרויות קיימות.
- איזו החלטה אנושית נדרשת.
```

למה זה עובד: הוא נותן לכלי רשות לעצור במקום לנחש החלטות מוצר, אבטחה או scope.
